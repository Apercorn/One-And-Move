# One-and-Move Implementation Plan

## Goal Description
Implement the core features for the crowdsourced transit and ride-request application:
1. User onboarding with social login and transport preferences.
2. A crowdsourcing engine for riders to report vehicle locations and fullness.
3. A ride request and recommendation engine to hybridize JUTC transit and robot taxis.
4. Real-time ETA calculations.
5. Map UI with custom routing polylines and markers.
6. Push notifications and alerts.

## Proposed Changes

### Database Layer (`packages/db`)
Add new schema files and update the index to export them.

#### [NEW] `packages/db/src/schema/transport.ts`
Define the tables for routes, vehicles, and crowdsourced reports:
- `preferred_transport` table: linked to `user_id`. (FR-01)
- `route` table: `id`, `name`, `polyline`. (FR-02)
- `vehicle` table: `id`, `type` (JUTC, robot_taxi), `official_number`, `current_route_id`. (FR-02)
- `vehicle_report` table: `id`, `vehicle_id`, `reporter_id`, `lat`, `lng`, `fullness`, `speed`, `reported_at`. (FR-02, FR-04)

#### [MODIFY] `packages/db/src/index.ts`
Export the new schemas.

#### [MODIFY] `packages/auth/src/index.ts`

### API Layer (`packages/api`)
Create tRPC/ORPC routers for the new endpoints.

#### [NEW] `packages/api/src/routers/vehicles.ts`
- `reportStatus`: Mutation for contributors to send GPS and fullness updates. Validates official number against DB. (FR-02)
- `getNearby`: Query to fetch live JUTCs and robot taxis. (FR-03)

#### [NEW] `packages/api/src/routers/routes.ts`
- `getBestRoute`: Query that returns hybrid transit combinations. Based on destination, it calculates routes using a mix of nearby taxis, buses, and short walking distances. Uses formula `(time * 0.5) + (crowding inverse * 0.3) + (cost * 0.2)` to return sorted recommendations containing ETA, fullness, cost, and markers. (FR-03)
- ETA Calculation logic embedded here, falling back to contributor speed if track points are missing. (FR-04)

#### [MODIFY] `packages/api/src/index.ts`
Merge the new routers into the main application router.

### Native App (`apps/native`)
Install required Expo SDK packages and implement the mobile-specific mapping and tracking.

#### [MODIFY] `apps/native/package.json`
Add `react-native-maps`, `expo-location`, `expo-task-manager`, `expo-notifications`, `expo-battery`, and `lucide-react-native` (for modern iOS icons). (FR-02, FR-05, FR-06)

#### [NEW] `apps/native/app/(tabs)/map.tsx`
The primary map view. Includes `MapView`, `Polyline`, and `Marker` for vehicles and arrival bubbles.

#### [NEW] `apps/native/lib/location-tracking.ts`
Background location tracking logic using `TaskManager` to send reports every 10 seconds to the `reportStatus` API.

#### [NEW] `apps/native/lib/notifications.ts`
Configure `expo-notifications` for push alerts. Check `expo-battery` to trigger the "Low Battery Mode" warning for contributors.

## Verification Plan

### Automated Tests
- N/A initially; the focus will be on feature implementation and manual verification first, as the application relies heavily on real-time location and maps which are difficult to unit-test comprehensively without extensive mock setups. 

### Manual Verification
1. **Database:** Generate and push the DB migrations using `drizzle-kit`. Verify the tables exist in Neon.
2. **Auth:** Ensure the login buttons for Google/Apple/Facebook render and correctly point to the auth endpoints.
3. **API & Engine:** Use the Swagger/OpenAPI docs generated by ORPC to manually send a `reportStatus` mutation, then verify that `getBestRoute` correctly weighs and returns that data.
4. **Native App:** Start the Expo dev server, open in iOS Simulator.
   - Verify Map renders with default location.
   - Verify "Board Bus" button requests location permissions.
   - Trigger a simulated location change in the simulator and ensure the API is called.
5. **Web App:** Start Next.js server, verify Mapbox renders with a polyline and custom icon markers.
